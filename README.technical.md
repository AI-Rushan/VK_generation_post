# Техническая документация: Генератор постов для товаров

Расширенное описание архитектуры, компонентов и поведения приложения для разработчиков.

## 1. Назначение

Приложение автоматизирует создание продающих постов для соцсетей по URL товара:

1. Извлекает данные со страницы товара.
2. Формирует управляемый промпт.
3. Запрашивает генерацию текста у LLM через ProxyAPI.
4. Публикует/планирует пост во ВКонтакте.
5. Ведет локальную историю избранных и запланированных постов.

## 2. Технологический стек

- Python 3.10+
- Flask 3 (`flask`)
- HTTP-клиент `requests`
- HTML-парсинг `beautifulsoup4`
- Jinja2-шаблоны (в составе Flask)

Зависимости: `requirements.txt`.

## 4. Конфигурация окружения

### 4.1 Переменные

- `PROXYAPI_KEY` - ключ доступа к ProxyAPI.
- `VK_ACCESS_TOKEN` - токен сообщества VK для публикации.
- `VK_GROUP_ID` - ID сообщества VK (число без `-`).

Пример: `.env.example`.

### 4.2 Загрузка `.env`

В `app.py` используется локальная функция `load_env_file()`:

- читает `.env` построчно;
- пропускает комментарии/пустые строки;
- устанавливает переменные через `os.environ.setdefault`.

Это означает, что уже заданные системные переменные не перезаписываются значениями из файла.

## 5. Архитектура и ключевые модули

### 5.1 `parse_product_page(product_url)`

Назначение: извлечь контекст товара.

- Выполняет `GET` страницы с браузерным `User-Agent`.
- Извлекает:
  - `<title>`,
  - `meta name="description"`,
  - текст из `h1/h2/p/li` (фильтр по длине >= 30 символов).
- Нормализует пробелы и ограничивает контент до 5000 символов.
- Возвращает `source_status` (`ok`, `http_<code>`, `fetch_error`).

### 5.2 `extract_title_from_url(product_url)`

Fallback при недоступной странице:

- берет сегмент пути URL;
- очищает slug (`-123`, `-` -> пробел);
- возвращает человекочитаемый заголовок.

### 5.3 `generate_post_with_llm(form_data)`

Формирование и отправка запроса к LLM:

- Проверяет наличие `PROXYAPI_KEY`.
- Читает настройки формы (тон, длина, эмодзи, CTA, хештеги, модель).
- Добавляет случайную эмоцию из списка `EMOTIONS` для вариативности.
- Собирает:
  - `system_prompt` (роль редактора),
  - `user_prompt` (ограничения по фактам, структуре и стилю).
- Отправляет запрос в `https://api.proxyapi.ru/openai/v1/chat/completions`.
- Возвращает `choices[0].message.content`.

Модели ограничены `MODEL_OPTIONS` (whitelist-подход).

### 5.4 `clean_post_text(text)`

Постобработка для удаления артефактов модели:

- удаляет нумерацию вида `1)`/`1.` в начале блоков;
- удаляет заголовки секций (`Заголовок`, `CTA`, `Хештеги` и т.д.);
- сокращает лишние пустые строки.

### 5.5 `publish_to_vk(text, publish_at=None)`

Интеграция с VK API (`wall.post`):

- публикует в сообщество через `owner_id=-<VK_GROUP_ID>`;
- использует `from_group=1`;
- при наличии `publish_at` отправляет `publish_date` для отложки;
- обрабатывает ошибки VK API, включая специальный сценарий `error_code=8`.

## 6. HTTP-слой и сценарии

Единственный маршрут: `@app.route("/", methods=["GET", "POST"])`.

Поддерживаемые действия формы:

- `generate` - генерация текста;
- `publish` - мгновенная публикация;
- `schedule` - отложенная публикация по `datetime-local`;
- `favorite` - сохранение поста в избранное;
- `save_edit` - сохранение правок из textarea.

Ключевая особенность UI: поле `force_action` позволяет принудительно отправлять `save_edit` из JS-логики кнопки редактирования.

## 7. Хранение данных

Используются JSON-файлы:

- `favorites.json` - избранные посты (до 30 последних).
- `scheduled_posts.json` - локальная история отложенных (до 20 последних).

Формат чтения/записи:

- `read_json_file(path)` возвращает `[]`, если файла нет или JSON поврежден;
- `write_json_file(path, data)` сохраняет UTF-8 JSON с `indent=2`.

## 8. Frontend

Файл: `templates/index.html`.

Содержит:

- форму настроек генерации;
- textarea результата;
- блоки избранного и отложенных постов;
- JS-функции:
  - copy to clipboard,
  - toggle edit/save для текста.

Стили: `static/styles.css`.

## 9. Обработка ошибок

- Ошибки HTTP-запросов к товарной странице и LLM ловятся как `requests.RequestException`.
- Невалидный ответ модели (`KeyError/IndexError/TypeError/ValueError`) обрабатывается отдельным сообщением.
- Ошибки времени отложенной публикации (`ValueError`) обрабатываются при парсинге datetime.
- Валидация обязательных полей выполняется перед каждым действием.

## 10. Локальный запуск

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
python app.py
```

Сервер разработки запускается на `127.0.0.1:5500` с `debug=True`.

## 11. Ограничения текущей реализации

- Нет БД и транзакционной модели; JSON-хранилище не подходит для high-load.
- Нет фонового воркера для надежного lifecycle управления отложенными задачами.
- Нет аутентификации/авторизации пользователей.
- Нет retry/backoff политики для внешних API.
- Нет централизованного логирования и метрик.

## 12. Рекомендации для production

- Вынести приложение за WSGI (Gunicorn/uWSGI) + reverse proxy.
- Перейти с JSON на PostgreSQL.
- Добавить очередь задач (Celery/RQ + Redis).
- Добавить `python-dotenv` и схемную валидацию env-конфига.
- Добавить structured logging и мониторинг ошибок.
- Разделить `app.py` на модули (routes/services/integrations/repositories).

## 13. Возможные точки расширения

- Поддержка нескольких каналов публикации (Telegram, OK, Facebook и т.д.).
- Мультиязычная генерация постов.
- Вариативная генерация (несколько версий поста за один запрос).
- Автотесты:
  - unit-тесты нормализации текста,
  - контрактные тесты интеграций,
  - e2e на пользовательские сценарии.
